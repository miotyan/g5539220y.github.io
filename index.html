2018-01-21

Scala是一个多写一个分号就是多余的艺术语言

所有Scala编译后的文件都是 *.class 后缀，也就是说，它完全运行在Java虚拟机中。

一切的开始—定义变量

1.变量用var定义就是可变的，相当于java中的普通变量。用val定义是不可变的，相当于java中final类型定义的变量

2.变量所有的类型不必声明，由值来决定

3.Scala有7种数值类型，和Java一样，但是没有包装类型

程序的灵魂—分支语句

Java中的if else 和elseif 完全适用于Scala中,if else的判断后可以用一个值来接，一共有两种方式

1.一句撸

2.块表达式

重复的工作—循环

Scala中有for和while、do while循环

在scala中循环变量的迭代用to表示

for循环

1. <- 相当于=，把迭代中的值赋给i

2. 如需定义多个变量，每个变量之间有一个”;” 在最后判断语句中不用

3. yield的作用是。。每迭代一次，就把i的值取出来做yield后面的操作

Scala的烧脑之处—函数与方法

函数在Scala中占着重要的地位，函数与方法类似，都是传值进去然后进行操作。不同的是，函数可以作为一个变量来使用

定义函数

定义函数的关键在于 => 这也是判断一个函数和方法的重要依据。函数的返回值可以省略（递归中不可省略），变量也可以省略（匿名函数）

定义方法

定义方法的关键在于 def和= 。方法中返回值也可以省略，方法不能作为变量传入，只能调用

方法与函数的结合(烧脑部分)

程序首先定义了一个传入一个函数的方法，方法的内容就是往传入的这个函数里面写两个Int值

接着再定义一个函数，内容是将传入得到两个参数相加

最后把定义好的函数传入方法中！

下划线的妙用

符号 _ 可以将方法转化为函数（原方法不变）也可以将系统中println等转化成函数

数的集合—元组、数组、集合、映射

数据和映射以及集合都包括定长和变长，只是两个包的事，就是

映射是特殊的对偶元组，元组的下角标是从1开始的，元组里可以存多种类型

4.List

5.Set

kafka生产者和消费者官网demo的小白分析

2018-01-11

生产者：

Properties JDK自带的设置参数类。一切的开始！

参数列表：

  1. bootstrap.servers 通过一个地址找到集群。尽量多写几个，防止一个不管用

  2. acks 用于ack-fail机制的配置

  3. retries 出错重传次数。要用这个就必须把max.in.flight.requests.per.connection设置为1

  4. batch.size producer 每次发送给同一个partition的request封装的（batch）大小.(一个partition有多个batch，一个batch对应一个partition)
  5. linger.ms producer 发送时会把两次间隔时间短的request聚合在一起，间接提高吞吐量。这个参数就是设置这个间隔时间的，间隔时间越大，发送的request数量就越少，但是延迟高！
  6. buffer.memory producer 自身的缓存内存大小。这个值主要涉及到数据发送问题，如果数据生产速度大于发送速度，就堵了。
  7. key.serializer和value.serializer 它俩的序列化类。直接照抄就行。

producer是Producer的实现类KafkaProducer把刚才那一堆参数都传进去而构造的

发送消息直接通过ProducerRecord把消息封装（topic信息，key分区信息，消息体）

发完一关！~ 生产者就这样。

消费者

参数列表：

  1. bootstrap.servers 同上
  2. group.id consumer的groupid
  3. enable.auto.commit 允许偏移量是否为自动提交
  4. auto.commit.interval.ms consumer向zookeeper提交offset的频率
  5. key.serializer和value.serializer 也是它俩的序列化类。直接照抄就行。

consumer和producer的构造类似，也是通过配置文件构造对象，同样通过KafkaConsumer

调用subscribe方法并传入一个asList来设置接收的各种话题

consumer通过调用poll方法取消息队列中的的队首同时移除该队列然后返回一个ConsumerRecords

遍历这个ConsumerRecords 取出各种值

zookeeper的个人理解和傻瓜式安装

2017-12-21

啥是zookeeper

zookeeper是一个集群分布式协调软件.

zookeeper自己就是一个集群，提供少量的数据的存储和管理以及提供对数据节点的监听、锁、同步功能可以把各个节点之间的数据同步，来保证一致性.

为啥会有zookeeper

假设有三个应用，这三个应用共用一份配置信息。

没有zookeeper的情况下，把配置信息放在一个服务器上,
并设定一把锁，在每次读写时只能一个应用来读写,
当这个服务器宕机时候，还需要另一个服务器做备用.

这时，这几台服务器的数据同步问题，锁的问题，以及其他问题。都很难解决。
zookeeper就是解决这些麻烦的问题。

当有zookeeper时，把这几个服务器
装上zookeeper!并把配置文件放在zookeeper里面就可以了

zookeeper的集群角色

Leader（主节点）与Follower（从节点）
Leader控制写的操作，超过一半的节点更新成功，就确认更新成功。
在启动时候没有Leader与Follower的角色。通过选举来确定Leader

Hbase的理解要点

2017-12-21

Hbase： Hadoop database 的简称，也就是基于Hadoop数据库，是nosql数据库，主要用于查询和显示各种数据

与Hive的区别：Hive则是Hadoop数据仓库，主要是写SQL来计算和处理HDFS上的结构化数据，用于离线的海量数据计算。

hbase也是基于hdfs存储的。它集群中的角色是：

主：HMaster 从：HRegionnserver

HMaster不存任何数据，只负责感知管理regionserver。
和负载均衡当集群启动，表创建完毕 HMaster挂了也没事

Hbase的表结构：

建表时不需要限定表中的字段，只需要指定若干列族！

插入数据时，列族中可以存储任意多个列！(列名(key)，值(value))

一个value可以有多个版本，通过版本号来确认，通常版本号就是一个时间戳

要查询一个具体字段的值，就需要指定坐标：表名—行键—-列族（ColumnFamliy）：列名（Qualifier）—版本

Hbase的存储机制：

把表切开存放，把某一个列族（store）下面的若干行切出region

region和region之间在hbase的集群当中管理，集群中有好多regionserver。region就在regionserver中

切分成region_table_1文件存在hdfs中，所以datanode和regionserver建议在一台机器上运行

可以减少很多网络传输问题

Hbase的寻址机制：

Hbase有个系统表（META表）。记录了一个region起始的行键的值到终止行键的值作为行键

内容为region文件在哪个机器上的ip地址.
一个region就在这个系统表中产生记录.

这个表的体积大到一定程度也需要再分.
再分出的信息还需要一个表（ROOT表）来存储它的行键范围和文件地址数据

ROOT表也在某一个regionserver中，具体位置应该在zookeeper上查

一个用户的数据表会被切分成很多region，然后分别存在很多不同的regionserver上再存在HDFS上
manstore 是一个存储访问量多的数据

想怎么输出就怎么输出----自定义OutputFormat

2017-12-18

自己定义OutputFormat输出，再也不用按照规定输出什么part-1-000000 以及什么一大串看不懂的CRC文件

原理分析：当reduce输出时，先调用OutputFormat里面的RecordWriter在writer出去。只要把Outputformat重写了，就能搞定了

写一个类继承了FileOutputFormat，kv泛型就是最终结果输出泛型。这里让文件输出到HDFS的自建文件夹，自建文件下

先在FileOutputFormat的子类里把文件输出流路径传给RecordWriter里，然后再传一个对象给Recordwrite的子类，再通过里面的write方法输出，然后关流。

以上解决后就可以把这个类写在程序中的main方法里了。

对于 FileOutputFormat.setOutputPath(job, new Path(args[1]));还是要写的，除了要输出文件外还需要输出success文件等其他在这个定义的目录里。
